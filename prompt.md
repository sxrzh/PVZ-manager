我们用 Rust 开发一个 Windows 小工具，用来给植物大战僵尸游戏的存档文件做版本管理，带有图形界面，使用 egui 库。

## 一些信息
- 本程序在 Windows 上运行，运行产生的所有数据文件保存在某个路径 SAVE_PATH=“C:\ProgramData\PVZ-manager” 下，这个路径用一个全局变量保存，便于日后修改；本程序的所有数据文件（除备份的游戏存档外）保存在这个文件夹下的 manager.json 中
- 游戏存档文件位于 C:\ProgramData\PopCap Games\PlantsVsZombies\userdata\，有 users.dat, user1.dat, user2.dat, ..., game1_51.dat, game1_60.dat, game1_69.dat 等，都是二进制文件，我们也不需要解析；
- 游戏中可以有多个用户，用户编号从 1 开始，users.dat 是用户总表，我们暂时不修改；user[n].dat 是用户 [n] 的元信息，也不修改；
- 游戏由多个小游戏组成，小游戏编号从 0 开始，每个用户有多个小游戏的存档，game[n]_[m].dat 是用户 [n] 的第 [m] 号小游戏的存档，这是我们要维护的主要内容。

## 功能

- 如果第一次运行，先让用户输入自己的用户编号，并保存到文件 manager.json
- 假设用户编号为 [n]，我们需要对每个 game[n]_[m].dat 维护一棵版本树，版本树上每个节点是一个存档备份或当前存档，根节点表示没有这个文件，每个节点的文件是由其父节点文件修改而来的，当前存档是一个特殊的节点，由某个备份节点或根节点修改而来，但还没有被备份（如果 userdata 中没有对应小游戏的当前存档则没有当前节点）；
- 有一个搜索框，输入对应小游戏名称或编号可以查看对应的版本树（已知的小游戏名称与编号的对应关系我已经写到本文件夹下的 game_id.json 中，程序应该读取这个文件）
- 每个存档备份有自己的名字，是用户可以修改的，备份文件存储在 SAVE_PATH 路径下，文件名应为 game[n]_[m]_backup[xxx].dat，[xxx] 是节点编号，程序将每个节点的元数据（名字、父节点编号、创建日期、备注）存储在 manager.json 的 nodes[]
- 考虑到部分用户可能觉得树状图过于复杂，显示分为表格模式和树形模式两种模式，默认是表格模式
    - 表格模式按时间顺序列出当前保存的所有存档的编号、名字、创建日期时间、备注，点击（不是右键点击）每一项后会出现弹窗，有详细信息（名字、日期时间、备注、由xx备份修改而来（即父节点））和按钮：恢复到此备份；删除此备份。按钮字要比较大；
        点击”恢复到此备份“后先弹窗确认，确认后将对应备份文件复制到 userdata 的 game[n]_[m].dat 覆盖，并以此节点作为当前节点的父节点
        点击”删除此备份“后先弹窗确认，确认后删除对应备份文件，但是**千万不要删除**节点信息及其子树；当且仅当一个节点子树中所有的文件都被删除，才删除这个节点的信息
    在表格模式的表格中，不显示当前存档和文件已被删除的节点
    - 树形模式以树的形式显示版本树，包括文件已被删除但子树还存在的节点；每个节点点击后出现和上一条一样的弹窗。根节点、有文件节点、被删除节点、当前存档节点以不同颜色显示。显示需要保证树边不相交，但是**不需要**支持拖动节点等没用的几何功能
- 无论表格模式还是树形模式，都有一个按钮”备份当前存档“，点击可以将当前存档节点提升为有文件节点，将 userdata 下的 game[n]_[m].dat 复制到 SAVE_PATH 下并改名作为这个新节点的备份文件
- 当前存档节点和根节点不能被改名和删除
- manager.json 分为 current 和 data 两个字段，current data 是一个列表，表示每个用户的所有节点，每一项又是一个对象，包括 roots（一个 map，存每个小游戏编号对应根节点编号），nodes（一个列表，每一项是一个节点的信息，当前存档节点不在这个表中），cur（一个整数，表示当前节点由哪个备份节点修改而来，如果没有当前存档则为 -1；初始化时，设为这个小游戏的根节点）

你可以先实现一个命令行工具，等命令行工具完善后再加入界面，当程序以参数 `-c` 运行时是命令行模式，支持节点级别的操作
